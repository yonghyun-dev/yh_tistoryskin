기존 프로젝트에 새로운 기능을 추가하는 작업이다.
바로 코드를 길게 작성하기보다, 먼저 기존 구조와 동작을 파악한 뒤 진행한다.

## 기능 추가 프로세스

### 1단계: 기존 코드 이해

- 프로젝트 구조를 확인한다
- 추가할 기능과 관련된 기존 파일을 찾는다
- 기존 코드의 패턴을 파악한다:
  - 라우터/컨트롤러 패턴
  - 서비스 레이어 패턴
  - DB 접근 패턴
  - 에러 처리 패턴
  - 테스트 패턴
- 기존 패턴을 따라 기능을 추가해야 한다

### 2단계: 추가 기능 분석

유저의 요구사항을 기반으로:

```markdown
### 추가 기능 분석

**기능명**: [이름]
**목적**: [이 기능이 왜 필요한지]

**기능 상세**:
| # | 세부 기능 | 설명 |
|:--|:--|:--|
| 1 | [세부기능] | [설명] |

**데이터 흐름**:
[사용자 입력] → [어디를 거쳐서] → [최종 결과]

**필요한 변경**:
| 종류 | 파일/대상 | 변경 내용 |
|:--|:--|:--|
| 수정 | [기존 파일 경로] | [어떤 수정] |
| 신규 | [새 파일 경로] | [역할] |
| DB | [테이블/컬렉션] | [스키마 변경 내용] |
| API | [엔드포인트] | [HTTP메서드 + 설명] |
| 환경변수 | [변수명] | [용도] |
```

### 3단계: 영향 범위 확인

| 확인 항목 | 영향 여부 |
|:--|:--|
| 기존 API 응답 형식 변경 | [있음/없음] |
| 기존 DB 스키마 변경 | [있음/없음] |
| 기존 기능 동작 변경 | [있음/없음] |
| 새 의존성 추가 | [있음/없음] |
| 환경변수 추가 | [있음/없음] |

기존 기능이 깨질 수 있는 지점이 있으면 **먼저 유저에게 알린다**.

### 4단계: 구현 계획 제시

- 수정 파일과 신규 파일을 구분해서 제안한다
- 구현 순서를 제시한다 (의존성 순서)
- 유저 승인 후 구현을 시작한다

### 5단계: 구현

- 기존 구조와 패턴을 **최대한 유지**하고, 필요한 부분만 최소 범위로 확장한다
- 한 번에 하나의 파일/기능을 완성한다
- 기존 코드와 스타일을 일치시킨다:
  - 같은 네이밍 컨벤션
  - 같은 에러 처리 패턴
  - 같은 import 스타일
  - 같은 주석 스타일

### 6단계: 완료 보고

```markdown
## 기능 추가 완료

### 추가된 기능
- [기능 설명]

### 변경/생성된 파일
| 파일 | 종류 | 내용 |
|:--|:--|:--|
| [경로] | 신규/수정 | [한 줄 설명] |

### API 변경 (있으면)
| 메서드 | 경로 | 설명 |
|:--|:--|:--|
| POST | /api/xxx | [설명] |

### DB 변경 (있으면)
| 테이블 | 변경 내용 |
|:--|:--|

### 테스트 방법
- [어떻게 동작을 확인하는지]

### 확인 필요 사항
- [있으면 기재]
```

---

## 원칙

- 과설계하지 말고, 현재 구조 안에서 가장 단순하게 확장한다
- 기존 API 응답 형식, DB 스키마, 폴더 구조는 명시하지 않으면 함부로 바꾸지 않는다
- 불확실한 내용은 "[확인 필요]"로 표시한다
- 승인 전에는 불필요한 전면 리팩토링이나 구조 변경을 하지 않는다

## 문서화

- 기능 추가 시 필요하면 docs 파일도 함께 제시한다
- 문서는 중학생도 이해할 수 있게 쉽게 작성한다
- 필요한 경우 Mermaid 다이어그램을 포함한다
- 작은 기능이면 문서는 생략 가능하다
