# 기능 추가 영향 분석 가이드

/add-feature에서 새 기능 추가 전 영향 범위를 파악할 때 참고한다.

---

## 1. 영향 분석 체크리스트

### 백엔드 영향

- [ ] 새 API 엔드포인트가 필요한가
- [ ] 기존 API 응답 형식이 변경되는가
- [ ] DB 스키마 변경이 필요한가 (테이블 추가/컬럼 추가)
- [ ] 마이그레이션이 필요한가
- [ ] 새 환경변수가 필요한가
- [ ] 새 외부 의존성(라이브러리)이 필요한가
- [ ] 기존 서비스/유스케이스를 수정해야 하는가
- [ ] 새 서비스/유스케이스를 만들어야 하는가
- [ ] 인증/인가 규칙이 변경되는가
- [ ] 기존 테스트가 깨질 수 있는가

### 프론트엔드 영향

- [ ] 새 페이지/라우트가 필요한가
- [ ] 새 컴포넌트가 필요한가
- [ ] 기존 컴포넌트를 수정해야 하는가
- [ ] 새 API 호출이 필요한가
- [ ] 상태 관리에 새 상태가 필요한가
- [ ] 새 타입 정의가 필요한가
- [ ] 네비게이션/메뉴가 변경되는가

### 인프라 영향

- [ ] 새 환경변수가 필요한가
- [ ] 배포 설정 변경이 필요한가
- [ ] 새 외부 서비스 연동이 필요한가
- [ ] 성능에 영향을 줄 수 있는가

---

## 2. 파일 분류 가이드

기능 추가 시 파일을 3가지로 분류한다:

| 분류 | 설명 | 예시 |
|:--|:--|:--|
| **신규 생성** | 이번에 새로 만드는 파일 | 새 라우터, 새 컴포넌트, 새 모델 |
| **수정 필요** | 기존 파일에 코드 추가/변경 | 라우터 등록, 타입 추가, 네비 메뉴 |
| **수정 금지** | 건드리면 안 되는 파일 | 다른 기능의 핵심 로직, 설정 파일 |

---

## 3. 기존 패턴 따르기 체크리스트

새 코드를 작성하기 전에 기존 코드의 패턴을 확인한다:

- [ ] 파일명 규칙 (snake_case? camelCase?)
- [ ] 폴더 위치 (같은 종류의 파일이 어디에 있는지)
- [ ] import 스타일
- [ ] 에러 처리 패턴
- [ ] 로깅 패턴
- [ ] 테스트 파일 위치와 네이밍
- [ ] 주석/docstring 스타일
- [ ] API 응답 형식
