요구사항을 반영해서 코드를 수정한다.
상세 디버깅 방법론은 `.claude/skills/fix/references/debugging-guide.md`를 참고한다.

## 수정 프로세스

### 0단계: 버그 분류

먼저 버그의 종류와 긴급도를 판단한다:

| 분류 | 예시 | 긴급도 |
|:--|:--|:--|
| 크래시/500 에러 | 서버 다운, 화이트스크린 | 즉시 |
| 데이터 손상 | 금액 0원, 중복 생성 | 즉시 |
| 보안 | 인증 우회, 데이터 노출 | 즉시 |
| 기능 오작동 | 필터 안 됨, 정렬 역순 | 높음 |
| 성능 | 5초+ 응답 | 보통 |
| UI | 레이아웃 깨짐 | 낮음 |
| 엣지 케이스 | 빈 문자열, 특수문자 | 낮음 |

### 1단계: 현황 파악

수정 대상을 정확히 이해한다.

- 유저의 요구사항에서 "현재 상태"와 "원하는 상태"를 구분한다
- 에러 메시지가 있으면 **정확히 읽는다** (타입, 파일, 라인, 스택 트레이스)
- 수정 대상 파일의 현재 코드를 읽는다
- 관련 파일(import하는 곳, import되는 곳)을 확인한다
- 기존 테스트가 있으면 확인한다
- 최근 변경 사항을 확인한다 (`git log --oneline -10`)

### 2단계: 영향 분석

수정이 다른 곳에 영향을 줄 수 있는지 확인한다.

| 확인 항목 | 방법 |
|:--|:--|
| 이 함수를 호출하는 곳 | Grep으로 검색 |
| 이 모듈을 import하는 곳 | Grep으로 검색 |
| API 응답 형식 변경 여부 | 프론트엔드/클라이언트에 영향 |
| DB 스키마 변경 여부 | 마이그레이션 필요 여부 |
| 환경변수 변경 여부 | 배포 설정 변경 필요 여부 |

영향 범위가 크면 유저에게 먼저 알리고 승인을 받는다.

### 3단계: 수정 방향 정리

각 수정 항목마다 아래를 정리한 뒤 수정한다:

```markdown
### [수정 항목명]

**현재 상태**: [지금 어떻게 동작하는지]
**문제**: [뭐가 문제인지]
**수정 방향**: [어떻게 바꿀지]
**영향 범위**: [어디에 영향 주는지]
**주의할 점**: [수정 시 조심할 것]
```

### 4단계: 수정 실행

- 한 번에 하나의 수정만 진행한다
- 수정 후 기존 기능이 깨지지 않았는지 확인한다
- 관련 테스트가 있으면 실행한다

### 5단계: 수정 완료 보고

```markdown
## 수정 완료

### 수정한 파일
| 파일 | 수정 내용 |
|:--|:--|
| [경로] | [한 줄 설명] |

### 변경된 동작
- Before: [이전 동작]
- After: [수정 후 동작]

### 확인 필요 사항
- [있으면 기재]
```

---

## 수정 원칙

- 기존 구조를 최대한 유지한다
- 요구사항에 없는 부분은 임의로 바꾸지 않는다
- 수정 범위를 최소화한다
- 백엔드/DB/API 변경 여부를 명확히 구분한다
- "이왕 고치는 김에" 식의 추가 수정은 하지 않는다
- 수정 전에 기존 코드가 왜 그렇게 작성되었는지 먼저 이해한다
- 불확실한 내용은 "[확인 필요]"로 표시한다

## 금지 사항

- 요구사항에 없는 리팩토링
- 관련 없는 파일 수정
- 수정 이유를 설명하지 않고 코드만 바꾸는 것
- 영향 범위 확인 없이 수정하는 것
