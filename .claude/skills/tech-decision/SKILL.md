프로젝트의 기술 아키텍처를 유저와 함께 결정한다.
요구사항이 확정된 후, 구현에 들어가기 전에 실행한다.

## 기술 결정 프로세스

### 1단계: 프로젝트 특성 파악

먼저 프로젝트의 규모와 특성을 파악한다:

| 질문 | 왜 중요한지 |
|:--|:--|
| 사용자 수가 어느 정도인가? | 인프라 규모, DB 선택에 영향 |
| 데이터가 얼마나 쌓이는가? | DB 필요 여부, 종류 결정 |
| 실시간성이 필요한가? | WebSocket, SSE, 폴링 중 선택 |
| 외부 서비스 연동이 있는가? | API 설계, 에러 핸들링 전략 |
| 배포 환경이 정해져 있는가? | Docker, 클라우드, 온프레미스 |

---

### 2단계: 카테고리별 기술 결정

아래 카테고리를 순서대로 검토한다. 각 카테고리에서 유저에게 선택지를 제시하고 결정을 받는다.

---

#### A. 개발 환경

| 선택지 | 적합한 경우 | 부적합한 경우 |
|:--|:--|:--|
| **venv (Python)** | 혼자 개발, 빠른 시작, 간단한 프로젝트 | 팀 개발, 환경 차이 문제 |
| **Docker** | 팀 개발, 복잡한 의존성, DB/Redis 등 인프라 포함 | 혼자 쓰는 간단한 프로젝트 |
| **Docker Compose** | 백엔드+프론트+DB+캐시 등 멀티 서비스 | 단일 서비스 |
| **nvm/volta (Node.js)** | 프론트엔드 단독 또는 Node.js 백엔드 | - |

**결정 기준**: 
- 혼자 개발 + 단순 프로젝트 → venv로 충분
- 팀 개발 or 인프라 포함 → Docker Compose 권장
- "나중에 Docker로 전환"은 비용이 낮으니 MVP는 venv로 시작해도 됨

---

#### B. 데이터 저장소

| 선택지 | 적합한 경우 | 부적합한 경우 |
|:--|:--|:--|
| **없음 (stateless)** | 이력 저장 불필요, 단순 API | 데이터 누적 필요 |
| **SQLite** | 혼자 사용, 소규모, 파일 기반 | 동시 접속, 대규모 |
| **PostgreSQL** | 실제 서비스, 복잡한 쿼리, JSONB | 단순 MVP |
| **MySQL** | 실제 서비스, 범용 | PostgreSQL 대비 기능 부족 |
| **MongoDB** | 스키마 유연성, 문서 기반 | 관계 데이터, 트랜잭션 필요 |
| **Supabase** | Auth+DB+Storage 통합, 빠른 개발 | 커스텀 제어 필요 |
| **Firebase** | 모바일/웹 실시간, Auth 통합 | 복잡한 쿼리 |
| **Redis** | 캐시, 세션, 실시간 | 영구 저장소 (보조로 사용) |

**결정 기준**:
- 이력 저장 불필요 → 없음
- 혼자 사용 + 이력 저장 → SQLite
- 실제 서비스 → PostgreSQL (기본) or MongoDB (문서 기반)
- 빠른 프로토타이핑 → Supabase

---

#### C. AI/ML 전략

| 선택지 | 적합한 경우 | 부적합한 경우 |
|:--|:--|:--|
| **단순 API 호출** | 범용 작업, 빠른 개발, 프롬프트로 충분 | 도메인 특화 지식 필요 |
| **RAG (검색 증강 생성)** | 자체 문서/데이터 기반 답변 필요 | 데이터가 없거나 적음 |
| **Graph RAG** | 복잡한 관계 데이터, 다단계 추론 | 단순 검색으로 충분 |
| **Fine-tuning** | 특정 스타일/형식 강제, 대량 학습 데이터 보유 | 데이터 부족, 빈번한 변경 |
| **온톨로지** | 도메인 지식 체계 구축, 의미 기반 검색 | 과설계, 구축 비용 높음 |
| **Embedding + 벡터 DB** | 유사도 검색, 의미 기반 매칭 | 단순 키워드 검색으로 충분 |
| **에이전트 (LangChain/CrewAI)** | 복잡한 멀티스텝 작업, 도구 사용 | 단순 질의응답 |

**결정 기준**:

```
요구사항이 프롬프트로 해결 가능한가?
├─ 예 → 단순 API 호출
└─ 아니오 → 자체 데이터가 있는가?
    ├─ 예 → 데이터 간 관계가 복잡한가?
    │   ├─ 예 → Graph RAG 또는 온톨로지
    │   └─ 아니오 → RAG (벡터 DB)
    └─ 아니오 → 특정 출력 형식이 필요한가?
        ├─ 예 → Fine-tuning
        └─ 아니오 → 프롬프트 엔지니어링으로 재시도
```

**RAG가 필요한 신호**:
- "우리 회사 문서를 기반으로 답변해야 해"
- "레시피 DB에서 검색해서 추천해야 해"
- "최신 정보가 반영되어야 해"

**RAG가 불필요한 신호**:
- "AI가 알아서 추천해주면 돼"
- "일반적인 지식으로 충분해"
- "외부 데이터 없이 사진만 분석하면 돼"

#### D. 인증/인가

| 선택지 | 적합한 경우 | 부적합한 경우 |
|:--|:--|:--|
| **없음** | 혼자 사용, 내부 도구 | 외부 사용자 있음 |
| **JWT** | API 서버, SPA | 서버 사이드 렌더링 |
| **세션 기반** | 전통적 웹앱, SSR | SPA, 마이크로서비스 |
| **OAuth2 (소셜 로그인)** | 사용자 편의, 빠른 가입 | 내부 도구 |
| **Supabase/Firebase Auth** | 빠른 구현, 관리 부담 감소 | 커스텀 인증 필요 |

---

#### E. 캐싱 전략

| 선택지 | 적합한 경우 | 부적합한 경우 |
|:--|:--|:--|
| **없음** | MVP, 요청 빈도 낮음 | 반복 요청 많음 |
| **인메모리 (dict/lru_cache)** | 단일 서버, 간단 | 서버 재시작 시 소실 |
| **Redis** | 다중 서버, 세션, 실시간 | 단일 서버 MVP |
| **CDN** | 정적 자산, 이미지 | 동적 API 응답 |

---

#### F. 배포

| 선택지 | 적합한 경우 | 부적합한 경우 |
|:--|:--|:--|
| **로컬 실행** | MVP, 개인 사용 | 외부 접근 필요 |
| **Vercel** | 프론트엔드, Next.js | 백엔드 (제한적) |
| **Railway/Render** | 풀스택, 간편 배포 | 대규모 트래픽 |
| **AWS (EC2/ECS)** | 프로덕션, 확장성 | 비용, 복잡성 |
| **Docker + VPS** | 커스텀 제어, 비용 효율 | 관리 부담 |

## 원칙

- 각 결정마다 **선택지 + 이유 + 대안**을 제시한다
- 유저가 잘 모르면 **프로젝트 특성에 맞는 것을 추천**한다
- 과설계를 경계한다 — MVP에 Graph RAG, 온톨로지는 대부분 불필요
- "나중에 바꿀 수 있는 결정"과 "지금 잘 정해야 하는 결정"을 구분한다
- 불확실한 내용은 "[확인 필요]"로 표시한다
