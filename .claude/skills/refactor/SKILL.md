이번 작업은 기존 프로젝트를 이어서 작업하거나 리팩토링하는 작업이다.
바로 코드를 수정하지 말고, 먼저 현재 프로젝트 구조와 핵심 파일을 파악한 뒤 시작한다.

## 리팩토링 프로세스

### 1단계: 현황 파악

- 프로젝트 폴더 구조를 훑고 전체 레이아웃을 이해한다
- 핵심 진입점 파일을 찾는다 (main, app, index 등)
- 주요 모듈의 역할을 파악한다
- 현재 동작 방식을 추정한다
- 기존 코드의 패턴과 컨벤션을 파악한다

### 2단계: 문제 진단

아래 관점에서 현재 코드의 문제점을 분류한다:

| 분류 | 확인 사항 |
|:--|:--|
| **구조 문제** | 레이어 위반, 순환 의존성, 과도한 결합 |
| **중복** | 같은 로직이 여러 곳에 반복 |
| **과설계** | 불필요한 추상 클래스, 구현체 1개인 인터페이스, 쓰이지 않는 패턴 |
| **응집도** | 관련 없는 코드가 한 파일에 섞여 있거나, 관련 있는 코드가 흩어져 있음 |
| **네이밍** | 역할을 파악할 수 없는 이름 |
| **에러 처리** | 누락된 에러 처리, 보여주기식 try-catch |
| **성능** | N+1 쿼리, 불필요한 전체 조회, 동기 블로킹 |

### 3단계: 리팩토링 계획

발견한 문제를 우선순위로 정리한다:

| 우선순위 | 기준 |
|:--|:--|
| **즉시 수정** | 버그 위험, 보안 문제, 데이터 정합성 |
| **이번에 수정** | 현재 작업과 직접 관련, 수정 비용 낮음 |
| **다음에 수정** | 관련 있지만 범위가 크거나 위험 |
| **보류** | 현재 작업과 무관, 동작에 문제 없음 |

각 수정 항목에 대해:
- 현재 상태 (뭐가 문제인지)
- 수정 방향 (어떻게 바꿀지)
- 영향 범위 (수정하면 어디가 영향받는지)
- 위험도 (수정 중 깨질 수 있는 것)

### 4단계: 유저 승인

- 리팩토링 계획을 유저에게 제시한다
- 유저가 범위를 조정하면 반영한다
- **승인 전에는 코드를 수정하지 않는다**

### 5단계: 실행

- 승인받은 범위 내에서만 수정한다
- 한 번에 하나의 리팩토링만 진행한다
- 각 수정 후 기존 기능이 깨지지 않았는지 확인한다
- 수정 내용을 기록한다

---

## 리팩토링 원칙

- **기존 기능을 깨뜨리지 않는다** — 이게 최우선
- 기존 패턴을 먼저 따르고, 문제가 명확할 때만 변경한다
- 과설계를 제거하는 리팩토링은 OK, 과설계를 추가하는 리팩토링은 금지
- 한 번에 너무 많이 바꾸지 않는다 — 점진적 개선
- 불확실한 내용은 "[확인 필요]"로 표시한다
- 유저가 요청하지 않은 영역은 건드리지 않는다
- "이왕 하는 김에" 식의 추가 수정은 하지 않는다

## 금지 사항

- 승인 전 대규모 구조 변경
- 동작하는 코드를 "더 깔끔하다"는 이유만으로 전면 재작성
- 관련 없는 파일까지 손대는 것
- 테스트 없이 핵심 로직 수정
