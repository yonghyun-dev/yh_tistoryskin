이번 작업은 빈 폴더에서 새 프로젝트를 시작하는 작업이다.
바로 코드를 작성하지 말고, Plan 모드를 활용해서 먼저 계획을 세운다.

## 프로세스

### 1단계: 요구사항 탐색

유저가 만들고 싶은 것을 이해하기 위해 질문한다.

#### 질문 순서

**1차 질문 (핵심 파악)**
- "어떤 서비스를 만들고 싶은가요? 한 줄로 설명해 주세요."
- "가장 중요한 기능 3개는 뭔가요?"
- "누가 사용하는 서비스인가요?"

**2차 질문 (범위 파악)** — 1차 답변 기반
- "웹인가요? 모바일인가요? 둘 다인가요?"
- "참고하는 서비스가 있나요?"
- "기존에 사용 중인 기술 스택이 있나요?"
- "혼자 개발인가요? 팀인가요?"

**3차 질문 (제약 파악)** — 필요한 경우에만
- "데이터베이스 선호가 있나요?"
- "배포 환경이 정해져 있나요?"
- "외부 서비스 연동이 필요한가요?"

질문 원칙:
- 한 번에 3개 이하
- 유저가 이미 구체적이면 질문을 줄인다
- 유저가 "잘 모르겠다"면 추천한다

### 2단계: 요구사항 정리

유저의 답변을 바탕으로 정리한다:

```markdown
## 프로젝트 개요
- 프로젝트명: [이름]
- 한 줄 설명: [설명]
- 대상 사용자: [누구]

## 핵심 기능
| # | 기능 | 설명 | 우선순위 |
|:--|:--|:--|:--|
| F-1 | [기능명] | [설명] | 필수 |
| F-2 | [기능명] | [설명] | 필수 |
| F-3 | [기능명] | [설명] | 선택 |

## 기술 스택 (추천)
| 분류 | 기술 | 이유 |
|:--|:--|:--|
| 언어 | [Python/TypeScript] | [이유] |
| 백엔드 | [FastAPI/Express] | [이유] |
| 프론트 | [React/Next.js] | [이유] |
| DB | [PostgreSQL/MongoDB] | [이유] |

## 미확정 사항
- [아직 결정되지 않은 것]
```

### 3단계: 프로젝트 계획

#### A. MVP 범위 정의
- 1차 완성 시 "무엇이 동작해야 하는지"를 명확히 한다
- 없어도 MVP가 동작하는 기능은 제외한다
- "있으면 좋겠다"는 백로그로 분류한다

#### B. 스프린트 분해
- Sprint 0: 프로젝트 초기 설정 (구조, 설정, DB, 인증)
- Sprint 1~N: 기능별 구현
- 각 스프린트에 완료 기준을 명시한다

#### C. 프로젝트 구조 설계

```
[프로젝트명]/
├── backend/          또는 바로 app/
│   ├── app/
│   │   ├── domain/
│   │   ├── usecases/
│   │   ├── adapters/
│   │   └── infrastructure/
│   ├── tests/
│   └── requirements.txt
├── frontend/         (필요 시)
│   └── src/
├── docs/
│   ├── requirements.md
│   ├── sprint.md
│   ├── agents/
│   └── context/
├── .claude/
│   ├── skills/
│   └── agents/
├── CLAUDE.md
├── .env.example
├── .gitignore
└── README.md
```

- 프로젝트에 불필요한 폴더는 만들지 않는다
- 구조는 유저의 기존 선호와 기술 스택에 맞게 조정한다

#### D. PRD 초안 작성

필요한 경우 `docs/prd.md`에 작성:

```markdown
# PRD: [프로젝트명]

## 1. 개요
## 2. 목표
## 3. 사용자 시나리오
## 4. 핵심 기능 상세
## 5. 화면/API 흐름 (Mermaid)
## 6. 데이터 모델
## 7. 개발 로드맵
## 8. 미확정 사항
```

### 4단계: 유저 승인

- 요구사항, 구조, 스프린트 계획을 유저에게 제시한다
- 수정 사항을 반영한다
- 승인 후 구현을 시작한다

---

## 원칙

- 지금은 코드보다 계획과 구조를 먼저 만든다
- 승인 전에는 구현 코드를 길게 작성하지 않는다
- 과설계하지 말고 MVP 기준으로 정리한다
- 문서는 중학생도 이해할 수 있게 쉽게 쓴다
- 필요한 경우 Mermaid 다이어그램을 포함한다
- 유저가 기술 선택에 확신이 없으면 추천하되, 이유를 설명한다
