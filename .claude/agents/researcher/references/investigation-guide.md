# 코드베이스 조사 가이드 (상용 수준)

researcher agent가 조사 시 따르는 체계적 가이드.

---

## 1. 조사 깊이 기준

| 수준 | 설명 | 사용 시점 | 소요 시간 |
|:--|:--|:--|:--|
| **Quick** | 폴더 구조 + 진입점만 파악 | "이 프로젝트가 뭐하는 거야?" | 2~3분 |
| **Standard** | 핵심 파일 분석 + 의존성 맵 | "이 기능 어디를 고쳐야 해?" | 5~10분 |
| **Deep** | 전체 코드 리딩 + 데이터 흐름 추적 | "리팩토링 영향 범위 파악" | 15~30분 |

---

## 2. 조사 순서

### 2.1 Quick Scan (항상 먼저)

1. **프로젝트 루트 확인**
   - `package.json` / `pyproject.toml` / `requirements.txt` → 기술 스택 파악
   - `README.md` → 프로젝트 목적
   - `.env.example` → 외부 의존성 파악
   - `docker-compose.yml` → 인프라 구성

2. **디렉토리 구조 스캔**
   - `find . -type f -name "*.py" -o -name "*.ts" | head -50` 수준
   - 폴더 이름으로 아키텍처 추론 (domain, usecases, adapters, api 등)

3. **진입점 찾기**
   - Python: `main.py`, `app.py`, `manage.py`, `wsgi.py`
   - Node.js: `package.json`의 `main`/`scripts.start`
   - 설정: `settings.py`, `config.ts`, `.env`

### 2.2 Standard Analysis (기능 조사 시)

4. **관련 파일 탐색**
   - Grep으로 키워드 검색 (함수명, 클래스명, URL 경로)
   - import 추적 (이 파일을 누가 import하는지)
   - 관련 테스트 파일 확인

5. **파일별 분석**
   각 파일에 대해:
   ```
   - 파일 경로:
   - 역할 (한 줄):
   - 주요 함수/클래스: (이름, 라인, 역할)
   - 의존성: (import 목록)
   - 사용처: (이 파일을 import하는 곳)
   - 현재 상태: (정상 / 문제 있음 / [확인 필요])
   ```

6. **의존성 맵 작성**
   ```
   A → B → C
   A → D
   D → C (순환 의존성 주의)
   ```

### 2.3 Deep Analysis (리팩토링/영향 범위 분석 시)

7. **데이터 흐름 추적**
   - 입력이 어디서 시작되어 어디로 가는지
   - 각 단계에서 데이터가 어떻게 변환되는지
   - 에러가 발생하면 어떻게 전파되는지

8. **기술 부채 식별**
   - TODO/FIXME/HACK 주석 검색
   - 중복 코드 패턴
   - 과도한 복잡도 (깊은 중첩, 긴 함수)
   - 테스트 없는 핵심 로직

## 3. 조사 대상별 체크리스트

### 3.1 새 프로젝트 분석

- [ ] 기술 스택 (언어, 프레임워크, DB)
- [ ] 프로젝트 구조 (폴더 트리)
- [ ] 진입점 (메인 파일)
- [ ] 설정 방식 (환경변수, 설정 파일)
- [ ] 외부 의존성 (API, DB, 캐시)
- [ ] 테스트 존재 여부 및 프레임워크
- [ ] CI/CD 파이프라인 존재 여부
- [ ] 문서화 수준 (README, docstring)

### 3.2 기능 영향 범위 분석

- [ ] 수정 대상 파일 목록
- [ ] 해당 파일을 import하는 모든 파일
- [ ] 해당 파일이 import하는 모든 파일
- [ ] 관련 API 엔드포인트
- [ ] 관련 DB 테이블
- [ ] 관련 테스트 파일
- [ ] 수정 시 깨질 수 있는 다른 기능
- [ ] 수정 금지 영역 (건드리면 안 되는 곳)

### 3.3 외부 API/라이브러리 조사

- [ ] 공식 문서 URL
- [ ] 인증 방식
- [ ] Rate Limit
- [ ] 응답 형식
- [ ] 에러 코드
- [ ] 무료/유료 구분
- [ ] 버전 호환성
- [ ] 대안 존재 여부

### 3.4 버그 원인 조사

- [ ] 에러 메시지 전문
- [ ] 에러 발생 파일과 라인
- [ ] 에러까지의 호출 스택
- [ ] 관련 입력 데이터
- [ ] 최근 변경 사항 (git log)
- [ ] 재현 조건
- [ ] 영향 범위

## 4. 보고서 품질 기준

### 필수 포함

- 조사 목적 (왜 이 조사가 필요한지)
- 파일 경로 + 라인 번호 (모든 참조에)
- 확인된 사실 vs 추측 구분 ("[확인 필요]" 표시)
- 다음 행동 제안

### 금지

- 코드를 읽지 않고 추측으로 보고
- 파일 경로 없이 "어딘가에 있을 것" 식 보고
- 너무 장황한 코드 복붙 (핵심만 발췌)
- 판단 없이 사실 나열만 (So what? 을 포함)
